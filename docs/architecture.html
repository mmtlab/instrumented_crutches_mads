Agent Structure

The system is built on MADS (Multi-Agent Distributed System) framework. Each component runs as an independent agent:

Master Crutch (Right) Agents

Agent Role Language Port
mads-broker Message hub, pub/sub router C++ 9092
coordinator Session management, command dispatch C++ custom
tip_loadcell_right Right tip force sensing C++ custom
eye_tracker Pupil Neon eye gaze streaming Python custom
ppg_right Right pulse oximetry (MAX30100) Python custom
hdf5_writer Data persistence to disk C++ custom
web_server HTTP interface, real-time web UI Python (FastAPI) 8000
status_handler System health monitoring C++ custom

Slave Crutch (Left) Agents

Agent Role Language
tip_loadcell_left Left tip force sensing C++
ppg_left Left pulse oximetry (MAX30100) Python

Data Pipeline

Sensor Data Flow

Hardware Sensor
    ↓
   ADC / I2C Interface
    ↓
   Agent (Read/Process)
    ↓
MADS Broker (Pub/Sub)
    ↓
   ├── Web Server (Real-time UI)
    ├── HDF5 Writer (Storage)
    └── Status Handler (Monitoring)
    ↓
   User (Download/Export)

Publisher/Subscriber Topics

Topic Publisher Data Rate Format

tip_loadcell_right tip_loadcell_right agent 1000 Hz JSON: {timestamp, force_N}
handle_loadcell_left left crutch agent 1000 Hz JSON: {timestamp, force_N}
handle_loadcell_right right crutch agent 1000 Hz JSON: {timestamp, force_N}
ppg_left ppg_left agent 25-100 Hz JSON: {timestamp, ir, red}
ppg_right ppg_right agent 25-100 Hz JSON: {timestamp, ir, red}
eye_tracker eye_tracker agent 200 Hz JSON: {timestamp, gaze_x, gaze_y, confidence}

Communication Protocol

MADS Networking

  • Protocol: ZeroMQ (ZMQ) with custom MADS framing
  • Pattern: Publisher/Subscriber with request/reply
  • Broker Role: Central message router at default port 9092
  • Reliability: Best-effort delivery; re-connections handled automatically

Time Synchronization

All timestamps are synchronized to microsecond precision:

  • Time Source: System clock on master, slave synchronizes via NTP or local calibration
  • Timestamp Format: Unix epoch (seconds.microseconds)
  • Synchronization Method: MADS broker provides common time reference
  • Drift Compensation: Post-acquisition linear interpolation if drift detected

Storage Model

HDF5 File Structure

Data is organized hierarchically in HDF5 format:

subject_01_session_01_run_01.hdf5
│
├── attributes/
│   ├── subject_id: "subject_01"
│   ├── session_id: "session_01"
│   ├── acquisition_date: "2026-03-20"
│   ├── start_timestamp: 1711000000.123456
│   └── end_timestamp: 1711003600.654321
│
├── metadata/
│   ├── master_ip: "10.42.0.1"
│   ├── mads_version: "2.0.0"
│   └── sensor_calibration/
│       ├── tip_left_scale: 0.0125
│       └── tip_right_scale: 0.0125
│
├── data/
│   ├── tip_loadcell_left/
│   │   ├── timestamp    [1M samples]
│   │   └── force        [1M samples]
│   ├── tip_loadcell_right/
│   │   ├── timestamp    [1M samples]
│   │   └── force        [1M samples]
│   ├── ppg_left/
│   │   ├── timestamp    [6M samples]
│   │   ├── ir           [6M samples]
│   │   └── red          [6M samples]
│   └── eye_tracker/
│       ├── timestamp    [1.2M samples]
│       ├── gaze_x       [1.2M samples]
│       ├── gaze_y       [1.2M samples]
│       └── confidence   [1.2M samples]
│
└── conditions/
    ├── timestamp        [segment markers]
    └── condition_name   [walking, turning, balancing...]

File Naming Convention

subject_{subject_id}_session_{session_id}_run_{run_number}.hdf5

Example:
subject_P001_session_001_run_01.hdf5
