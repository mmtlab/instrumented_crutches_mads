Startup Issues

Problem: Services don't start on boot

Symptoms: Web server not accessible at startup

Solution:

  1. Verify services are enabled: sudo systemctl status mads-broker
  2. Check service dependencies: systemctl show mads-web_server -p Before,After
  3. View boot logs: sudo journalctl -b -e
  4. Re-enable services: 
    sudo systemctl daemon-reload
sudo systemctl enable mads-broker.service
sudo systemctl enable mads-web_server.service

Problem: "Cannot connect to broker" error

Symptoms: Agents fail to start, broker connection timeout

Solution:

  1. Verify broker is running: sudo systemctl status mads-broker
  2. Check broker logs: sudo journalctl -u mads-broker -f
  3. Test broker connectivity: telnet localhost 9092
  4. Restart broker: 
    sudo systemctl restart mads-broker.service
sleep 2
sudo systemctl status mads-broker

Problem: Web server starts but interface is blank

Symptoms: Port 8000 responds but no content loads

Solution:

  1. Check web server logs: sudo journalctl -u mads-web_server -f
  2. Verify data directory exists: ls -la ~/data
  3. Ensure static files are present: ls -la web_server/static/
  4. Restart web server: 
    sudo systemctl restart mads-web_server.service
  5. Try with verbose logging: 
    cd ~/instrumented_crutches_mads/web_server
python web_server.py 2>&1 | tee debug.log

Network & Connectivity

Problem: Slave crutch not connecting to master

Symptoms: Status shows "disconnected" or no left crutch data

Solution:

  1. Verify network connectivity: ping {master_ip}
  2. Check IP configuration on slave: 
    ip addr show
ip route show
  3. Verify mads.ini broker URL: grep broker_url /usr/local/etc/mads.ini
  4. Test broker port from slave: nc -zv {master_ip} 9092
  5. Restart networking and agents: 
    sudo systemctl restart networking
sudo systemctl restart mads-tip_loadcell.service

Problem: Eye tracker not connecting

Symptoms: "Eye tracker disconnected" in status

Solution:

  1. Verify Pupil glasses are powered and within range
  2. Check WiFi connection: iwconfig or nmcli dev wifi show
  3. Verify glasses IP: arp -a (look for Pupil device)
  4. Restart eye tracker agent: sudo systemctl restart mads-eye_tracker.service
  5. View eye tracker logs: 
    sudo journalctl -u mads-eye_tracker.service -f

Sensor Problems

Problem: Load cell readings are zero or nonsensical

Symptoms: Forces show 0 N or incorrect values

Solution:

  1. Verify sensor calibration: grep -A 2 load_cell_calibration /usr/local/etc/mads.ini
  2. Check I2C communication: i2cdetect -y 1
  3. Physically test load cells with known weight
  4. Recalibrate if needed (see Configuration)
  5. Restart load cell agent: 
    sudo systemctl restart 'mads-*_loadcell*.service'

Problem: PPG sensor not responding (no red/ir values)

Symptoms: PPG data shows "NA" or error in web interface

Solution:

  1. Check I2C bus: i2cdetect -y 1 (should see device at 0x57)
  2. Verify PPG agent is running: sudo systemctl status mads-ppg_right.service
  3. Check agent logs: 
    sudo journalctl -u mads-ppg_right.service -f --no-pager
  4. If I2C not responding, verify physical connections (SDA/SCL) and pull-ups
  5. Reset PPG sensor by power cycling crutch

Problem: Load cell values drift over time

Symptoms: Baseline creeps despite calibration

Solution:

  1. This is normal thermal drift over long sessions
  2. Recalibrate periodically during acquisition: click "Calibrate" in web UI
  3. Increase ADC oversampling in configuration for smoother readings
  4. Ensure crutches are at room temperature before starting

Web Interface Issues

Problem: Cannot access web interface (connection refused)

Symptoms: curl: (7) Failed to connect

Solution:

  1. Verify web server is running: lsof -i :8000
  2. Check firewall: sudo ufw status
  3. Allow port 8000: sudo ufw allow 8000/tcp
  4. Restart web server: 
    sudo systemctl restart mads-web_server.service
sleep 2
curl http://localhost:8000/api/status

Problem: Real-time data visualization is slow or freezing

Symptoms: Charts stutter, browser CPU high

  1. Reduce sampling rates in configuration (lower load cell rate from 1000 to 500 Hz)
  2. Close unnecessary browser tabs
  3. Try different browser (Chrome recommended)
  4. Check CPU usage on master: top
  5. Reduce web server worker threads: edit service file and restart

Performance Issues

Problem: System gets slow during recording

Symptoms: High CPU/memory usage, card writes lag

Solution:

  1. Check disk space: df -h (ensure > 10% free)
  2. Monitor CPU: htop
  3. Verify microSD card speed (UHS-II recommended)
  4. Reduce sampling rates if not critical for experiment
  5. Ensure no other background processes are running: ps aux | grep -v grep

Problem: Data drops or missing samples

Symptoms: Gaps in recorded data, warning in logs

Solution:

  1. These may occur under extreme load
  2. Check system logs for out-of-memory errors: dmesg | grep -i memory
  3. Increase MADS broker buffer size in configuration
  4. Reduce total sampling rate if not critical

Data & Storage Issues

Problem: Cannot download sessions or export data

Symptoms: Download button disabled or times out

Solution:

  1. Check data directory: ls -lh ~/data/
  2. Verify disk space: df -h
  3. Check HDF5 file integrity: h5dump -H file.hdf5
  4. Restart web server: sudo systemctl restart mads-web_server.service
  5. Try manual download via SSH: scp pi@host:~/data/file.hdf5 .

Problem: HDF5 files are corrupt or cannot open

Symptoms: Python error when loading file

Solution:

  1. Verify file wasn't interrupted mid-write: check system logs during acquisition
  2. Check file size is reasonable for recording duration
  3. Try recovery with h5py: 
    python3 -c "import h5py; f = h5py.File('file.hdf5', 'r'); print(list(f.keys()))"
  4. If corrupt, file is likely lost (always use Stop button properly)

Emergency Procedures

Emergency Stop: If system becomes unresponsive: 
# Kill all MADS services
sudo systemctl stop 'mads-*.service'

# Force kill remaining processes
sudo killall -9 coordinator web_server hdf5_writer

# Revert to known state
sudo systemctl start mads-broker.service
sleep 2
sudo systemctl start 'mads-*.service'

For persistent issues: Collect logs and contact support with:

  • System logs: journalctl -e > /tmp/system.log
  • Conference output: systemctl list-units --failed
  • Hardware info: cat /proc/cpuinfo /proc/meminfo
